# manga-ocr CPU recognize — 스레드 배치 벤치마크 정리

작성 2026-07-06. 측정 도구: [bench_recognize_threads.py](bench_recognize_threads.py).

## 배경 — 왜 쟀나

- recognize 엔진(**manga-ocr**)은 `DEFAULT_DEVICE=cpu`, detector(comic-text-and-bubble-detector)도 CPU 기본. 즉 지금 기본 조합에서 **detect·recognize는 둘 다 CPU**에서 돈다. (GPU 기본은 PaddleOCR-VL 정도.)
- 파이프라인은 detect+recognize를 `gpu_lock`으로 직렬화하고, recognize는 이미지 안에서 **crop마다 순차**로 부른다. 즉 지금은 한 추론이 전 코어(intra-op)를 쓰고, 이미지들은 한 번에 하나씩.
- 질문 두 개:
  1. CPU recognize를 **한 추론에 전 코어(직렬)** 대신 **여러 싱글스레드 워커(병렬)** 로 돌리면 이득인가?
  2. 병렬화한다면 풀 크기를 **물리 코어** 기준으로? **논리 스레드(하이퍼스레드)** 기준으로?

## 측정 방법

- `(workers × threads-per-worker)` 배치를 여러 개 스윕, **같은 크롭 세트**를 워밍업(전 워커 모델 로드) 후 정상상태로 돌려 **crops/sec** 측정.
- **핀 고정**: `os.sched_setaffinity`(리눅스 표준, psutil 불필요). `/sys/.../topology`로 물리 코어 수를 정확히 읽고, 워커를 서로 겹치지 않는 CPU 슬라이스에 배치(`8w×1t` → 물리 코어 8개에 하나씩).
- 입력: 실제 페이지 21장 → 디텍터로 **42 크롭** (`--detect`).
- 환경: **8C16T**, `force_cpu`, 도커 컨테이너 안(`docker compose run`).
- 재현:
  ```bash
  docker compose run --rm -T \
    -v /경로/페이지폴더:/pages:ro \
    server python - --data /pages --detect \
    < packages/scanlation-server/tools/bench_recognize_threads.py
  ```

## 판독

1. **병렬이 이득.** 물리 코어당 싱글스레드 워커(`8w×1t`)가 지금 방식(`1w×전코어`)의 **~1.85배**. 한 추론에 16스레드 몰아주기는 스케일이 나빠(오히려 최하위권) 병렬만 못하다.

2. **물리 코어가 단위, 하이퍼스레드는 손해.** 순위가 `8w×1t(물리) > 8w×2t > 16w×1t`. 논리 스레드를 어떻게 쓰든(2t로 묶든, 16프로세스로 쪼개든) `8w×1t`(HT 미사용)보다 아래. SIMD/포트 공유 병목.
   - 참고: 하이퍼스레드를 *굳이 쓴다면* `8w×2t`(6.40) > `16w×1t`(5.87) — 모델 사본이 8개 vs 16개라 L3·메모리 압력이 절반이라 그렇다.

3. **반쯤 메모리 바운드.**
   - intra-op 효율: `1w×16t / 1w×1t` = 2.5배 / 16스레드 = **~16%** (메모리 + 작은 모델의 스레드 동기화 오버헤드 합작).
   - 멀티워커 효율: `8w×1t / 1w×1t` = 4.68배 / 8워커 = **~58%** (동기화 없는 독립 프로세스 기준). 완전 대역폭 포화면 0배여야 하는데 58% → **연산 병렬도 대역폭 한계까지는 실제로 먹힌다** = 반쯤 메모리 바운드.
   - 같은 8코어에서 `8w×1t`(독립) > `4w×2t`(intra-op) +20% → intra-op 동기화가 그만큼 비효율.

4. **핀은 무효 → 병목이 코어 배치가 아니라 메모리.** 핀 유/무 차이가 전부 노이즈(±2.5%). 배치가 병목이었다면 핀에서 숫자가 튀었을 텐데 안 튐 → 병목은 코어 배치보다 아래(메모리)임을 역으로 확정. 동시에 **핀-없음 수치가 이미 믿을 만했다는 검증**도 됨.

5. **최선 ~1.8~1.9배가 hard ceiling.** 핀으로도 안 오름 = 대역폭 천장. 적은 수치는 아니나 극적이지도 않다.

## 실용 판단 — 보류 (2026-07-06)

- **반영하려면** recognize를 **싱글스레드 워커 풀(≈물리 코어 수)** 로 재구성해야 함: 모델 사본 ×물리코어(메모리), 프로세스 IPC, 지금의 `gpu_lock` 단일 실행 구조 교체.
- **이득 ~2배 vs 복잡도 큼** → 지금은 보류. 벤치는 "병렬·물리 코어가 이득"이라는 답만 확보.
- **대안 후보(더 조사 가치 있음)**: crop들을 **한 forward에 배치(batch)** 하면 arithmetic intensity가 올라 대역폭 병목을 직접 깎는다(weight를 한 번 읽어 N crop에 재사용). 워커 풀보다 이게 나을 수 있음 — 단 manga-ocr API가 이미지 1장씩이라 배치 지원을 손봐야 함.
